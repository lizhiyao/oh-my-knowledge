# 知识质量度量体系——完整路线图

> 2026-04-04 制定，2026-04-05 更新（完整度拆分为两层）

## 理论框架

```
知识质量 = f(完整度, 准确度, 有效度)
```

| 维度 | 核心问题 | 度量方式 | 当前状态 |
|------|----------|----------|----------|
| **完整度** | "知识够不够？测试够不够？" | 知识完备性 + 用例覆盖率 | ⚠️ 用例覆盖率已实现，知识完备性空白 |
| **准确度** | "说的对不对？" | 事实性校验 | ❌ 空白 |
| **有效度** | "有没有用？" | 因果归因 | ⚠️ 仅有分数对比 |

### 完整度的两层含义

之前把完整度等同于"测评用例覆盖率"，这只是一半。完整度实际包含两个不同的问题：

| 层次 | 问题 | 类比 |
|------|------|------|
| **知识完备性** | 知识库本身有没有缺口？ | "教材编全了吗？" |
| **用例覆盖率** | 测试用例有没有覆盖到已有知识？ | "考试题出够了吗？" |

用例覆盖率 100% 不代表知识完备——可能教材本身就漏了一章。知识完备但用例覆盖率低也没意义——教材写全了但没考过，不知道学生学没学会。

两层都需要度量。

## Phase 1B：知识完备性评估（新增）

**目标**：评估知识库本身在其领域中的覆盖程度——有没有缺概念、缺实体、缺规则。

**回答的问题**：知识库声称覆盖售卖域，但售卖域的哪些页面/组件/场景在知识库里没有对应条目？

### 三种互补的评估方法

#### 方法一：Schema Diff（从代码仓库倒推）

**原理**：代码仓库是领域的 ground truth——有多少页面目录、多少组件、多少路由，是客观可枚举的。拿这个枚举和知识库（如 semantic-index.md）做 diff，就知道知识库缺了什么。

**做法**：
1. 扫描 cwd 下的代码结构（页面目录、组件目录、路由配置等）
2. 提取实体清单（页面名、组件名、模块名）
3. 和 semantic-index.md 中的关键词映射表做 diff
4. 输出：知识库已覆盖的实体 / 代码中存在但知识库未收录的实体

**产出**：
- `lib/schema-diff.ts`
- 新 CLI 命令 `omk bench audit --schema-diff`
- 输出 `SchemaDiffReport: { covered, uncovered, coverageRate }`

**示例输出**：
```
售卖域 Schema Diff:
  代码中共 45 个页面/组件目录
  semantic-index 覆盖 28 个（62%）
  未覆盖：
    - repos/example-app-app/src/pages/refund/     （退保页）
    - repos/components-insure-common-timer/   （倒计时组件）
    - ...
```

**实施难度**：低。只需要 fs 扫描 + 文本匹配。

#### 方法二：能力问题法（Competency Questions）

**原理**：来自本体工程领域。先定义知识库"应该能回答哪些问题"（能力问题），再检验它能不能回答。能力问题可以从 skill 内容用 LLM 自动生成。

**做法**：
1. 从 artifact 内容提取领域范围描述
2. 用 LLM 生成一组能力问题（"确认页有哪些组件？""投保按钮在哪个仓库？""被保人有几种样式？"）
3. 将能力问题作为 eval-samples，用 agent 跑一轮评测
4. 回答不了的 = 知识缺口

**产出**：
- 扩展 `gen-samples --competency`
- 生成的样本专门探测知识边界，断言聚焦"能否回答"而非"回答质量"
- 和 Schema Diff 互补——Schema Diff 找"缺了什么概念"，能力问题找"有概念但回答不了"

**实施难度**：中。核心是 LLM prompt 设计。

**学术来源**：[Competency Questions Survey](https://link.springer.com/chapter/10.1007/978-3-031-47262-6_3)、[Evaluating Ontologies with CQ](https://ieeexplore.ieee.org/document/6690745/)

#### 方法三：缺口信号采集（从评测输出倒推）

**原理**：agent 在实际执行中遇到知识缺口时，会产生可观测的信号——PRD skill 会标记【推断】【知识缺口】，agent 可能会搜索但找不到结果（Grep 返回空）。把这些信号结构化采集，就是知识缺口的直接证据。

**做法**：
1. 从 agent 输出中提取缺口标记（【推断】【知识缺口】【可能过期】）
2. 从 toolCalls 中提取失败的搜索（Grep 无结果、Read 文件不存在）
3. 聚合为知识缺口报告

**产出**：
- `lib/gap-detector.ts`
- 从 VariantResult.fullOutput + toolCalls 中自动提取缺口信号
- 写入 Report 的 `analysis.knowledgeGaps` 字段
- 报告详情页新增"知识缺口"区块

**示例输出**：
```
知识缺口信号：
  - [推断] 退保流程：semantic-index 无匹配（prd-refund-001）
  - [搜索无结果] Grep "倒计时|countdown" in semantic-index.md（prd-high-freq-hit）
  - [文件不存在] Read repos/example-app-app/src/pages/refund/（prd-refund-001）
```

**实施难度**：低。数据已在 toolCalls 和 fullOutput 中。

**学术来源**：[Don't Hallucinate, Abstain](https://arxiv.org/abs/2402.00367)、[Detecting Prompt Knowledge Gaps](https://arxiv.org/html/2501.11709v1)

### 三种方法的关系

```
Schema Diff（代码 → 知识库）  → "缺了哪些概念"（静态）
能力问题法（LLM 生成 → 评测）  → "有概念但回答不了"（动态）
缺口信号采集（评测输出 → 分析）→ "实际遇到了什么缺口"（实证）
```

三者从不同角度夹击：
- Schema Diff 是**自上而下**——从领域全景推导应有知识
- 能力问题法是**由内而外**——从知识库自身推导应能回答的问题
- 缺口信号采集是**自下而上**——从实际使用中发现的缺口

### Phase 1B 实施优先级

```
Schema Diff（低难度，最快出结果）
  ↓
缺口信号采集（低难度，数据已有）
  ↓
能力问题法（中难度，需要 LLM prompt 设计）
```

## Phase 2：事实性校验

**目标**：agent 输出中的事实声明，自动和知识库/代码交叉验证。

| 步骤 | 产出 | 原理 |
|------|------|------|
| 2.1 事实声明提取 | `lib/fact-extractor.ts` | 从输出中提取文件路径、模块名、能力状态等可验证声明 |
| 2.2 校验引擎 | `lib/fact-checker.ts` | 文件路径→existsSync，模块名→semantic-index 搜索，代码引用→Grep |
| 2.3 事实性报告 | 报告新区块 | ✓ 已验证 / ✗ 已反驳 / ? 无法验证 |

---

## Phase 3：有效度验证

**目标**：证明知识确实改善了 agent 行为，不只是"分数变了"。

| 步骤 | 产出 | 原理 |
|------|------|------|
| 3.1 因果归因 | `lib/attribution-analyzer.ts` | 对比 baseline vs skill 的执行轨迹，定位分叉点 |
| 3.2 消融实验 | `omk bench ablation` 命令 | 逐个移除知识，看退化幅度→知识重要度排名 |
| 3.3 一致性度量 | 扩展 `--repeat` | 多次运行看内容一致性（不仅是分数） |
| 3.4 反馈闭环 | `omk bench feedback import` | 真实使用反馈→新测试样本→知识改进 |
