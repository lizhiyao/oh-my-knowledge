# 知识质量度量体系——完整路线图

> 2026-04-04 制定，2026-04-05 更新（完整度拆分为两层）

## 理论框架

```
知识质量 = f(完整度, 准确度, 有效度)
```

| 维度 | 核心问题 | 度量方式 | 当前状态 |
|------|----------|----------|----------|
| **完整度** | "知识够不够？测试够不够？" | 知识完备性 + 用例覆盖率 | ⚠️ 用例覆盖率已实现，知识完备性空白 |
| **准确度** | "说的对不对？" | 事实性校验 | ❌ 空白 |
| **有效度** | "有没有用？" | 因果归因 | ⚠️ 仅有分数对比 |

### 完整度的两层含义

之前把完整度等同于"测评用例覆盖率"，这只是一半。完整度实际包含两个不同的问题：

| 层次 | 问题 | 类比 |
|------|------|------|
| **知识完备性** | 知识库本身有没有缺口？ | "教材编全了吗？" |
| **用例覆盖率** | 测试用例有没有覆盖到已有知识？ | "考试题出够了吗？" |

用例覆盖率 100% 不代表知识完备——可能教材本身就漏了一章。知识完备但用例覆盖率低也没意义——教材写全了但没考过，不知道学生学没学会。

两层都需要度量。

## Phase 1B：知识风险度量（新增）

**目标**：不是证明"知识是完整的"（这永远证明不了），而是量化知识的风险敞口——有多少场景可能踩到盲区？这个风险在收敛还是发散？

### 为什么不叫"知识完备性"

"完备性 85%"是一个静态指标，不能帮任何人做决策。不同角色需要不同层次的答案：

| 角色 | 核心问题 | omk 应提供的答案 |
|------|----------|-----------------|
| **Skill 作者** | "我改了有没有变好？" | 分数对比、断言详情、执行轨迹（P1A 已实现） |
| **域负责人** | "知识够不够用？该补哪块？" | 缺口清单、缺口分类、补充优先级 |
| **CTO** | "值不值得继续投？风险可控吗？" | 缺口率趋势、风险收敛曲线、投入产出比 |

omk 当前只服务了第一层。Phase 1B 要打通第二层和第三层。

### 核心指标：知识缺口率

```
知识缺口率 = 触发知识盲区的评测次数 / 总评测次数
```

这个数字比"覆盖率 85%"更有意义：
- 域负责人看到"缺口率 30%"，知道该补知识了
- CTO 看到"上月 30%，本月 18%，环比 -40%"，知道投入有回报
- 缺口率持续下降 = 知识工程在收敛 = 值得继续投

### 三个数据源（从易到难）

#### 数据源一：缺口信号采集（从评测输出中提取）

**原理**：agent 遇到知识盲区时会产生可观测信号——标记【推断】【知识缺口】，搜索返回空结果，读取文件不存在。这些信号已经在 toolCalls 和 fullOutput 中，只需要结构化采集。

**采集的信号**：
- agent 输出中的缺口标记（【推断】【知识缺口】【可能过期】）
- 失败的搜索（Grep 无结果）
- 文件不存在（Read 失败）
- agent 显式声明"没有找到相关信息"

**产出**：
- `lib/gap-detector.ts`
- 每次评测自动产出缺口清单
- 写入 Report 的 `analysis.knowledgeGaps` 字段
- 报告详情页新增"知识缺口"区块

**给域负责人看的**：
```
本次评测发现 3 个知识缺口：
  1. [搜索无结果] "倒计时|countdown" 在 semantic-index 中无匹配
     → 建议：补充倒计时相关组件到语义索引
  2. [文件不存在] repos/example-app-app/src/pages/refund/
     → 建议：clone 退保页仓库或补充退保知识
  3. [推断] 被保人第四种样式的切换逻辑
     → 建议：更新 design.md 补充样式切换条件
```

**给 CTO 看的**：
```
知识缺口率：3/6 = 50%（6 个评测任务中 3 个触发了知识盲区）
```

**实施难度**：低。数据已有。

#### 数据源二：跨评测趋势追踪

**原理**：单次评测的缺口率只是快照。真正有价值的是趋势——每次补了知识、跑一轮评测，缺口率是不是在降。

**做法**：
- 每次评测的缺口清单写入报告
- 趋势页展示缺口率随时间的变化曲线
- 标注关键事件（"补充了退保知识"→ 缺口率从 50% 降到 30%）

**产出**：
- 扩展现有趋势页（`/trends/:variant`），增加缺口率曲线
- 域负责人看到的是"哪些缺口被修复了、哪些还在"
- CTO 看到的是"这条曲线在往下走"

**实施难度**：中。需要跨报告聚合。

#### 数据源三：领域边界探测（主动发现盲区）

**原理**：前两个数据源都是"被动发现"——评测样本碰巧触及了盲区才能发现。这个数据源是"主动探测"——用 LLM 生成能力问题（Competency Questions），专门去试探知识边界。

**做法**：
1. 从 artifact 内容提取领域范围描述
2. 用 LLM 生成一组边界探测问题（"退保流程怎么走？""多币种怎么切换？""团险和个险的差异？"）
3. 用 agent 跑一轮评测，统计回答不了的比例
4. 回答不了的 = 知识边界之外的盲区

**产出**：
- `gen-samples --probe-boundaries`
- 自动生成的探测样本 + 探测报告
- 域负责人拿到一份"你的知识库在这些场景下会失灵"的清单

**实施难度**：中。核心是 LLM prompt 设计。

**学术来源**：
- 本体工程 [Competency Questions](https://link.springer.com/chapter/10.1007/978-3-031-47262-6_3)
- LLM 知识缺口 [Don't Hallucinate, Abstain](https://arxiv.org/abs/2402.00367)

### Phase 1B 三层产出

```
Skill 作者 → 不需要额外产出（P1A 已满足）

域负责人 → 缺口清单 + 补充建议 + 优先级
           （数据源一：每次评测自动产出）

CTO     → 缺口率 + 趋势曲线 + 投入产出比
           （数据源二：跨评测聚合）
```

### Phase 1B 实施优先级

```
缺口信号采集（低难度，数据已有，立即可做）
  ↓
跨评测趋势追踪（中难度，需要跨报告聚合）
  ↓
领域边界探测（中难度，需要 LLM prompt 设计）
```

## Phase 2：事实性校验

**目标**：agent 输出中的事实声明，自动和知识库/代码交叉验证。

| 步骤 | 产出 | 原理 |
|------|------|------|
| 2.1 事实声明提取 | `lib/fact-extractor.ts` | 从输出中提取文件路径、模块名、能力状态等可验证声明 |
| 2.2 校验引擎 | `lib/fact-checker.ts` | 文件路径→existsSync，模块名→semantic-index 搜索，代码引用→Grep |
| 2.3 事实性报告 | 报告新区块 | ✓ 已验证 / ✗ 已反驳 / ? 无法验证 |

---

## Phase 3：有效度验证

**目标**：证明知识确实改善了 agent 行为，不只是"分数变了"。

| 步骤 | 产出 | 原理 |
|------|------|------|
| 3.1 因果归因 | `lib/attribution-analyzer.ts` | 对比 baseline vs skill 的执行轨迹，定位分叉点 |
| 3.2 消融实验 | `omk bench ablation` 命令 | 逐个移除知识，看退化幅度→知识重要度排名 |
| 3.3 一致性度量 | 扩展 `--repeat` | 多次运行看内容一致性（不仅是分数） |
| 3.4 反馈闭环 | `omk bench feedback import` | 真实使用反馈→新测试样本→知识改进 |

## 竞品差异化

| 维度 | omk | DeepEval | agentevals | AgentBench |
|------|-----|---------|-----------|-----------|
| 输出质量 | ✅ | ✅ | ❌ | ✅ |
| 行为过程 | ✅ | ✅ | ✅ | ❌ |
| **用例覆盖率** | ✅ P1A | ❌ | ❌ | ❌ |
| **知识缺口率 + 趋势** | ✅ P1B | ❌ | ❌ | ❌ |

**没有人在做知识风险度量。** 所有竞品都在评"agent 做得好不好"，omk 还能回答"知识的风险敞口有多大、在不在收敛"——这是从工具到平台的关键跳跃。
