# 知识质量的两个根本性问题

> 2026-04-04 首次讨论，持续迭代

omk 当前能回答"改了知识，效果变好还是变坏"，但还没回答两个根本性问题。

## 问题一：知识完整度如何测？

完整度包含两层不同的问题：

### 第一层：测评用例覆盖率（已实现）

"测试够不够？"——当前测试样本触及了知识域中多少文件。

**状态**：Phase 1A 已实现。PRD skill 首次评测显示 5/25 = 20% 覆盖率。

### 第二层：知识完备性（核心难题）

"知识够不够？"——知识库本身在其领域中是否有缺口。

**核心矛盾**：你不知道你不知道什么。semantic-index 有 120 个关键词映射，但售卖域可能有 200 个概念——剩下 80 个根本没在知识库里。

**三种互补的评估方法**（2026-04-05 调研确定）：

1. **Schema Diff**（代码仓库 → 知识库）：扫描代码中有多少页面/组件，和 semantic-index 做 diff，找出"代码里有但知识库没收录"的实体。自上而下，静态分析。
   - 学术来源：知识图谱完备性评估（Schema Completeness）

2. **能力问题法**（LLM 生成 → 评测）：从 skill 内容自动生成"知识库应该能回答的问题"，跑评测看能否回答。由内而外，动态探测。
   - 学术来源：本体工程 Competency Questions

3. **缺口信号采集**（评测输出 → 分析）：从 agent 输出中提取【推断】【知识缺口】标记，从 toolCalls 中提取搜索无结果的信号。自下而上，实证驱动。
   - 学术来源：LLM 知识缺口检测（Don't Hallucinate, Abstain）

## 问题二：现有指标是否足以证明知识质量？

三层指标各有缺陷：

| 指标层 | 能力 | 缺陷 |
|--------|------|------|
| 规则断言（contains、regex、tools_called） | 确定性，可重复 | 只测"有没有"，不测"好不好" |
| LLM judge（rubric + 1-5 分） | 能评主观质量 | 黑盒，1-5 分尺度粗，换模型分数不同 |
| 行为指标（turns、toolCalls、成本） | 客观可量化 | 只测过程效率，不测结果正确性 |

**缺失维度**：

- 事实性验证：agent 说"确认页在 example-app-app/src/pages/confirm/"，这条路径真的对吗？目前没有 ground truth 校验
- 内容一致性：同一问题跑 5 次，答案内容（不仅是分数）是否一致
- 真实使用反馈：生产中的使用场景回流为 eval-samples，形成闭环

## 可做的三件事

1. **知识覆盖率**——从"你告诉我测什么"变成"我告诉你还缺什么"
2. **事实性校验**——agent 输出中的路径、模块名等可自动和代码交叉验证
3. **使用反馈闭环**——真实场景回流为测试用例
