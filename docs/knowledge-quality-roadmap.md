# 知识质量度量体系——完整路线图

> 2026-04-04 制定

## 理论框架

```
知识质量 = f(完整度, 准确度, 有效度)
```

| 维度 | 核心问题 | 度量方式 | 当前状态 |
|------|----------|----------|----------|
| **完整度** | "还有什么没测到？" | 知识覆盖率 | ❌ 空白 |
| **准确度** | "说的对不对？" | 事实性校验 | ❌ 空白 |
| **有效度** | "有没有用？" | 因果归因 | ⚠️ 仅有分数对比 |

---

## Phase 1：知识覆盖率

**目标**：评测完成后，自动输出知识覆盖率报告。

| 步骤 | 产出 | 原理 |
|------|------|------|
| 1.1 知识消费追踪 | `lib/knowledge-tracker.ts` | 从 toolCalls 提取 agent 读了哪些文件、搜了哪些关键词 |
| 1.2 知识索引解析 | `lib/knowledge-index.ts` | 从 skill 内容解析知识域边界（引用的文件、知识库目录） |
| 1.3 覆盖率计算 | `lib/coverage-analyzer.ts` | 文件覆盖率 + 搜索覆盖率 → 综合覆盖率 |
| 1.4 覆盖率可视化 | 报告新区块 | 热力图展示哪些知识被触及、哪些是盲区 |
| 1.5 自动补全样本 | `--cover-gaps` 命令 | LLM 基于未覆盖知识自动生成测试样本 |

**优先级最高**：基础设施已就绪，产出物对团队立即可见。

---

## Phase 2：事实性校验

**目标**：agent 输出中的事实声明，自动和知识库/代码交叉验证。

| 步骤 | 产出 | 原理 |
|------|------|------|
| 2.1 事实声明提取 | `lib/fact-extractor.ts` | 从输出中提取文件路径、模块名、能力状态等可验证声明 |
| 2.2 校验引擎 | `lib/fact-checker.ts` | 文件路径→existsSync，模块名→semantic-index 搜索，代码引用→Grep |
| 2.3 事实性报告 | 报告新区块 | ✓ 已验证 / ✗ 已反驳 / ? 无法验证 |

---

## Phase 3：有效度验证

**目标**：证明知识确实改善了 agent 行为，不只是"分数变了"。

| 步骤 | 产出 | 原理 |
|------|------|------|
| 3.1 因果归因 | `lib/attribution-analyzer.ts` | 对比 baseline vs skill 的执行轨迹，定位分叉点 |
| 3.2 消融实验 | `omk bench ablation` 命令 | 逐个移除知识，看退化幅度→知识重要度排名 |
| 3.3 一致性度量 | 扩展 `--repeat` | 多次运行看内容一致性（不仅是分数） |
| 3.4 反馈闭环 | `omk bench feedback import` | 真实使用反馈→新测试样本→知识改进 |

## 度量指标总表

| 指标 | 维度 | 阶段 | 目标值 |
|------|------|------|--------|
| 文件覆盖率 | 完整度 | P1 | ≥ 80% |
| 搜索覆盖率 | 完整度 | P1 | ≥ 60% |
| 综合覆盖率 | 完整度 | P1 | ≥ 70% |
| 事实验证率 | 准确度 | P2 | ≥ 80% |
| 事实反驳率 | 准确度 | P2 | ≤ 5% |
| 归因清晰度 | 有效度 | P3 | ≥ 60% |
| 知识重要度 | 有效度 | P3 | 关键知识 ≥ 0.5 分退化 |
| 内容一致性 | 有效度 | P3 | ≥ 85% |
